### 3.1 创建工作流 YAML

首先，我创建了一个工作流文件 `workflows/wordpress-theme-v2.yaml`：

```
# WordPress 主题 v2.0 开发工作流
name: WordPress Theme v2.0 Development
description: 参考 Beep 主题风格重新设计 CareerCompass 主题
version: 1.0

phases:
  - name: 阶段 1 - 设计
    parallel_tasks:
      - agent: design-ui-designer
        task: |
          基于 Beep 主题风格更新 CareerCompass Design Token
          
          要求：
          1. 字体改用等宽字体（Roboto Mono、Fira Code 等）
          2. 增加文章分类导航样式
          3. 优化卡片设计（粗边框、明显阴影）
          4. 单栏居中布局（max-width: 720px）
          5. 保留 CareerCompass 蓝紫配色（#667eea → #764ba2）
          
          输出：
          - JSON 格式 Design Token
          - 配色方案说明
          - 字体使用规范

      - agent: design-brand-guardian
        task: |
          待命审核 UI 设计师的 Design Token 方案
          
          审核要点：
          1. 是否保持 CareerCompass 品牌特色
          2. 蓝紫配色是否正确使用

          输出：
          - 审核报告
          - 改进建议
          - 品牌一致性评分

  - name: 阶段 2 - 开发
    depends_on: 阶段 1 - 设计
    parallel_tasks:
      - agent: engineering-frontend-developer
        task: |
          根据 Design Token 实现 WordPress 主题代码
          
          输入：
          - UI 设计师的 Design Token
          - 品牌守护者的审核意见
          
          任务：
          1. 更新 style.css（完整样式）
          2. 更新 header.php（添加分类导航）
          3. 更新 front-page.php（首页模板）
          4. 更新 index.php（文章列表）
          5. 添加 Google Fonts 支持
          
          输出：
          - 完整的主题文件
          - 代码说明文档

  - name: 阶段 3 - 审查
    depends_on: 阶段 2 - 开发
    parallel_tasks:
      - agent: engineering-code-reviewer
        task: |
          审查前端开发者提交的代码
          
          审查要点：
          1. WordPress 编码规范
          2. 代码安全性
          3. 性能优化
          4. 可访问性
          5. 响应式设计
          
          输出：
          - 审查报告
          - 问题列表（分级：阻塞/建议/小改进）
          - 修复建议

  - name: 阶段 4 - 文档
    depends_on: 阶段 3 - 审查
    parallel_tasks:
      - agent: engineering-technical-writer
        task: |
          更新主题 README 文档
          
          输入：
          - 最终的主题代码
          - 代码审查报告
          
          任务：
          1. 更新主题说明
          2. 添加 Design Token 文档
          3. 编写使用指南
          4. 更新更新日志
          
          输出：
          - README.md
          - 使用指南

quality_gates:
  - phase: 阶段 1 - 设计
    criteria: 品牌一致性评分 >= 90
  - phase: 阶段 3 - 审查
    criteria: 阻塞问题数量 = 0

success_metrics:
  - 设计符合 Beep 主题风格
  - 保持 CareerCompass 品牌特色
  - 代码符合 WordPress 规范
  - 无阻塞级别问题
  - 文档完整清晰
```

**工作流设计要点：**

- **阶段划分** – 设计→开发→审查→文档，每个阶段职责明确
- **并行任务** – UI 设计师和品牌守护者可以并行工作
- **依赖关系** – 开发依赖设计完成，审查依赖开发完成
- **质量门禁** – 品牌一致性≥90 分，阻塞问题=0 才能进入下一阶段
- **成功指标** – 明确定义什么是”完成”

## 五、对 agency-orchestrator 的优化建议

### 5.1 问题：一次性安装 186 个智能体

**现状：**

原始的 `install.sh` 会一次性安装所有 186 个智能体：

- 创建 558 个文件（每个智能体 3 个文件：SOUL.md、AGENTS.md、IDENTITY.md）
- 占用 10-20 MB 磁盘空间
- 管理困难，难以找到需要的智能体
- 学习曲线陡峭，新手容易懵

**我们的优化方案：**

编写 `install-selective.sh` 脚本，支持**按需安装**：

- 只安装当前任务需要的智能体
- 每次约安装 6 个智能体，18 个文件
- 清晰知道每个智能体的用途
- 可以随时添加新的智能体

### 5.2 install-selective.sh 脚本

**脚本地址：**[https://github.com/jnMetaCode/agency-agents-zh/blob/main/scripts/install-selective.sh](https://github.com/jnMetaCode/agency-agents-zh/blob/main/scripts/install-selective.sh)

**用法：**

```
# 安装指定的智能体
./scripts/install-selective.sh   design-ui-designer   engineering-frontend-developer   engineering-code-reviewer
```

**核心逻辑：**

```
#!/usr/bin/env bash

# 1. 检查 integrations/openclaw/ 是否存在
check_integrations() {
  if [[ ! -d "$INTEGRATIONS" ]]; then
    err "integrations/openclaw 不存在"
    err "请先运行：./scripts/convert.sh --tool openclaw"
    exit 1
  fi
}

# 2. 遍历指定的智能体名称
for agent_name in "$@"; do
  local src_dir="$INTEGRATIONS/$agent_name"
  local dest_dir="$OPENCLAW_AGENTS/$agent_name"
  
  # 3. 检查源目录是否存在
  if [[ ! -d "$src_dir" ]]; then
    err "智能体不存在：$agent_name"
    continue
  fi
  
  # 4. 复制 SOUL.md、AGENTS.md、IDENTITY.md
  cp "$src_dir/SOUL.md" "$dest_dir/SOUL.md"
  cp "$src_dir/AGENTS.md" "$dest_dir/AGENTS.md"
  cp "$src_dir/IDENTITY.md" "$dest_dir/IDENTITY.md"
  
  # 5. 注册到 OpenClaw
  openclaw agents add "$agent_name" --workspace "$dest_dir"
done

# 6. 备份 openclaw.json
backup_openclaw_config() {
  local config_file="${HOME}/.openclaw/config/openclaw.json"
  local backup_file="${HOME}/.openclaw/config/openclaw.json.backup.$(date +%Y%m%d%H%M%S)"
  cp "$config_file" "$backup_file"
}
```

## 六、经验教训

### 6.1 我犯过的错误

- **一开始啥都自己干** – 小弟们闲着，我累得要死
- **没有先创建工作流就开干** – 各自为战，没有协同
- **假装完成了** – 其实小弟还没干活，就想蒙混过关
- **需求确认不够** – 做了很多无用功，反复修改

### 6.2 正确的做法

- **创建工作流** – 定义 phases 和 tasks，明确每个小弟的职责
- **分配任务** – 让专业的人做专业的事
- **设置质量门禁** – ≥90 分通过，否则打回重做
- **跟踪进度** – 实时监控每个小弟的输出
- **汇总验收** – 我负责编排和验收，不负责具体执行

## 八、总结与展望

### 8.1 核心收获

- **别啥都自己干** – 指挥小弟干活，我负责编排和验收
- **先提问确认需求** – 再执行任务，避免无用功
- **用 PUA 要结果** – 不看过程看输出，用数据说话
- **每次修改后立即验证** – curl 测试，确保真的完成了
- **按需安装智能体** – 不要一次性安装 186 个，用多少装多少

### 8.2 可复用的方法论

- **智能体编排模式** – 设计→开发→审查→文档
- **质量门禁** – ≥90 分通过，否则打回重做
- **评分机制** – 奖惩分明，连续 3 次<80 分优化掉
- **PUA 技巧** – 不轻易放弃，穷尽一切解决方案
- **按需安装** – 按任务类型推荐智能体组合
