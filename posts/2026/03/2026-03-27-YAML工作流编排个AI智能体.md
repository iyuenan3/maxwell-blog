## 二、工作流 YAML 设计思路

### 2.1 阶段划分原则

我将工作流分为 4 个阶段：**设计→开发→审查→文档**。

**为什么是这 4 个阶段？**

- **设计** – 先明确视觉规范和品牌标准
- **开发** – 根据设计规范实现代码
- **审查** – 确保代码质量和规范
- **文档** – 记录使用说明和最佳实践

这个顺序符合软件开发的自然流程，每个阶段的输出是下一阶段的输入。

### 2.2 并行任务 vs 串行任务

**可以并行的任务：**

- UI 设计师 + 品牌守护者（设计阶段）- 品牌守护者可以提前准备审核标准
- 前端开发者 + 技术文档工程师（开发阶段）- 文档可以同步编写

**必须串行的任务：**

- 开发依赖设计完成 – 没有 Design Token 无法写代码
- 审查依赖开发完成 – 没有代码无法审查
- 文档依赖审查完成 – 确保文档基于最终代码

**判断标准：**是否有依赖关系，是否需要前序输出。

### 2.3 质量门禁设计

我设置了两个质量门禁：

**如何设置合理的门禁标准？**

- **太严（≥ 95 分）** – 导致反复修改，效率低
- **太松（≥ 80 分）** – 质量无法保证
- **合理（≥ 90 分）** – 平衡质量和效率

### 2.4 成功指标定义

我定义了 5 个成功指标：

- 设计符合 Beep 主题风格
- 保持 CareerCompass 品牌特色
- 代码符合 WordPress 规范
- 无阻塞级别问题
- 文档完整清晰

这些指标都是可衡量的，不是模糊的”做好”。”做好”是主观的，”90 分”是客观的。

### 3.1 工作流完整代码

<pre class="wp-block-preformatted"><code class="language-yaml"># WordPress 主题 v2.0 开发工作流
name: WordPress Theme v2.0 Development
description: 参考 Beep 主题风格重新设计 CareerCompass 主题
version: 1.0

phases:
  - name: 阶段 1 - 设计
    parallel_tasks:
      - agent: design-ui-designer
        task: |
          基于 Beep 主题风格更新 CareerCompass Design Token
          
          要求：
          1. 字体改用等宽字体（Roboto Mono、Fira Code 等）
          2. 增加文章分类导航样式
          3. 优化卡片设计（粗边框、明显阴影）
          4. 单栏居中布局（max-width: 720px）
          5. 保留 CareerCompass 蓝紫配色（#667eea → #764ba2）
          
          输出：
          - JSON 格式 Design Token
          - 配色方案说明
          - 字体使用规范

      - agent: design-brand-guardian
        task: |
          待命审核 UI 设计师的 Design Token 方案
          
          审核要点：
          1. 是否保持 CareerCompass 品牌特色
          2. 蓝紫配色是否正确使用
          3. 字体选择是否符合技术博客定位
          4. 整体风格是否统一
          
          输出：
          - 审核报告
          - 改进建议
          - 品牌一致性评分

  - name: 阶段 2 - 开发
    depends_on: 阶段 1 - 设计
    parallel_tasks:
      - agent: engineering-frontend-developer
        task: |
          根据 Design Token 实现 WordPress 主题代码
          
          输入：
          - UI 设计师的 Design Token
          - 品牌守护者的审核意见
          
          任务：
          1. 更新 style.css（完整样式）
          2. 更新 header.php（添加分类导航）
          3. 更新 front-page.php（首页模板）
          4. 更新 index.php（文章列表）
          5. 添加 Google Fonts 支持
          
          输出：
          - 完整的主题文件
          - 代码说明文档

  - name: 阶段 3 - 审查
    depends_on: 阶段 2 - 开发
    parallel_tasks:
      - agent: engineering-code-reviewer
        task: |
          审查前端开发者提交的代码
          
          审查要点：
          1. WordPress 编码规范
          2. 代码安全性
          3. 性能优化
          4. 可访问性
          5. 响应式设计
          
          输出：
          - 审查报告
          - 问题列表（分级：阻塞/建议/小改进）
          - 修复建议

  - name: 阶段 4 - 文档
    depends_on: 阶段 3 - 审查
    parallel_tasks:
      - agent: engineering-technical-writer
        task: |
          更新主题 README 文档
          
          输入：
          - 最终的主题代码
          - 代码审查报告
          
          任务：
          1. 更新主题说明
          2. 添加 Design Token 文档
          3. 编写使用指南
          4. 更新更新日志
          
          输出：
          - README.md
          - 使用指南

quality_gates:
  - phase: 阶段 1 - 设计
    criteria: 品牌一致性评分 >= 90
  - phase: 阶段 3 - 审查
    criteria: 阻塞问题数量 = 0

success_metrics:
  - 设计符合 Beep 主题风格
  - 保持 CareerCompass 品牌特色
  - 代码符合 WordPress 规范
  - 无阻塞级别问题

## 五、踩坑记录与解决方案

### 5.1 坑 1：并行任务变成串行

**问题：**UI 设计师和品牌守护者实际是串行执行。

**原因：**品牌守护者需要等待 UI 设计师输出才能审核。

**解决：**重新定义任务，品牌守护者可以先准备审核标准，而不是被动等待。

<pre class="wp-block-preformatted"><code class="language-yaml"># 优化前
- agent: design-brand-guardian
  task: 审核 UI 设计师的 Design Token 方案

# 优化后
- agent: design-brand-guardian
  task: |
    待命审核 UI 设计师的 Design Token 方案
    
    审核要点：
    1. 是否保持 CareerCompass 品牌特色
    2. 蓝紫配色是否正确使用
    3. 字体选择是否符合技术博客定位
    4. 整体风格是否统一
    
    输出：
    - 审核报告
    - 改进建议
    - 品牌一致性评分</code></pre>

### 5.2 坑 2：质量门禁太严格

**问题：**品牌一致性评分 ≥ 95 分，导致反复修改。

**影响：**设计阶段耗时从 30 分钟增加到 1 小时。

**解决：**调整为 ≥ 90 分，平衡质量和效率。

**经验：**完美是优秀的敌人。90 分足够好，95 分代价太高。

### 5.3 坑 3：任务描述不够清晰

**问题：**前端开发者理解错了 Design Token。

**表现：**用了浅色背景而不是深色。

**解决：**在任务描述中添加具体示例。

<pre class="wp-block-preformatted"><code class="language-yaml"># 优化前
task: 根据 Design Token 实现 WordPress 主题代码

# 优化后
task: |
  根据 Design Token 实现 WordPress 主题代码
  
  输入：
  - UI 设计师的 Design Token
  - 品牌守护者的审核意见
  
  任务：
  1. 更新 style.css（完整样式）
  2. 更新 header.php（添加分类导航）
  3. 更新 front-page.php（首页模板）
  4. 更新 index.php（文章列表）
  5. 添加 Google Fonts 支持
  
  输出：
  - 完整的主题文件
  - 代码说明文档</code></pre>

### 5.4 坑 4：缺少错误处理

**问题：**智能体执行失败时没有重试机制。

**影响：**需要手动重新触发。

**解决：**添加自动重试和失败通知。

<pre class="wp-block-preformatted"><code class="language-yaml"># 建议添加
error_handling:
  max_retries: 3
  retry_delay: 60  # 秒
  on_failure: notify  # 失败时通知</code></pre>

## 六、最佳实践总结

### 6.1 工作流设计原则

- **阶段划分要符合逻辑** – 设计→开发→审查→文档
- **并行任务要真正独立** – 没有依赖关系才能并行
- **质量门禁要合理** – ≥ 90 分，不要追求完美
- **成功指标要可衡量** – 不是”做好”，是”90 分”

### 6.2 任务描述技巧

- **明确输入和输出** – 智能体知道需要什么、交付什么
- **提供具体示例** – 避免理解偏差
- **定义验收标准** – 什么是”完成”
- **包含约束条件** – 时间、质量、资源限制

### 6.3 质量门禁设置

- 不要设置完美标准（≥ 95 分）
- 合理标准是 ≥ 90 分
- 阻塞问题必须 = 0
- 建议问题可以协商

### 6.4 可复用的 YAML 模板

## 七、给其他开发者的建议

### 7.1 什么时候需要工作流编排？

- 多个智能体协作（3 个以上）
- 任务有明确依赖关系
- 需要质量保证
- 需要可复用的流程

### 7.2 如何设计第一个工作流？

- 从简单开始（2-3 个阶段）
- 逐步增加复杂度
- 记录每次迭代
- 形成模板

### 7.3 常见误区

- 阶段划分太细（超过 6 个）- 增加管理成本
- 并行任务太多（实际是串行）- 没有真正节省时间
- 质量门禁太严（导致反复修改）- 效率低
- 任务描述太模糊（智能体理解错）- 输出不符合预期
